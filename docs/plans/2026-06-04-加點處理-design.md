# 設計：加點（追加品項）處理

**日期**：2026-06-04
**範圍**：CitrusKDS（Android）+ Compass_KDS（後端 .NET）

## 目標

當一筆已在 KDS 的訂單被「加點」（顧客追加品項，新品項以 `ItemStatus='j'` 進來，原品項已是 W/O）時：

1. KDS 卡片**顯示回「新單」**提示店員（資料庫的 W/O 品項維持原狀，不改回 J）。
2. **自動接單開** → 自動把新增的 j 品項 → W，並**自動印一張只含新增品項**的廚房單，**最上方標「加點」**。
3. **自動接單關** → 店員按「接單」時，才把 j→W 並印該加點單。
4. 同一筆單**再加點** → 出現新的 j，重複流程。
5. 連帶修正：接單/完成/取餐對**混合狀態**訂單只搬動對應來源狀態的品項。

## 偵測模型（無狀態，不需 session 快照）

依後端帶出的 **per-item `ItemStatus`** 判斷：

- **加點待處理** = 該單同時有 ≥1 個 `j/J`（未接）品項 **且** ≥1 個 `W/O`（已接）品項。
- **全新單** = 全部品項都是 `j/J`。
- **卡片顯示狀態優先序**：有任何 `j` → **新單(J)**；否則有 `W` → **製作中(W)**；否則有 `O` → **待取餐(O)**。
- **要印的加點品項** = 接單當下該單所有 `ItemStatus IN ('j','J')` 的品項。印完轉 W，下次輪詢不再被當加點（自然去重）。

> DB 事實佐證：新加品項在 `OrdersItem` 為獨立列，`ItemStatus='j'`（小寫）、較晚 `CreateDate`、較大 `OrderSeq`。`IsPrint` 未可靠維護，不採用。

## 後端改動（Compass_KDS）

### 1. Detail 帶出 per-item 狀態
- `OrdersDetail` 新增 `ItemStatus`（必要）、（選配）`OrderSeq`。
- `OrdersQueryData` 已有 `ItemStatus`；main 查詢已 SELECT/GROUP BY `I.[ItemStatus]` → 只需在組裝迴圈 map 進 `OrdersDetail.ItemStatus`。
- served/recall 查詢的 SELECT/GROUP BY 補上 `I.[ItemStatus]`（目前 served/recall 的 OrdersData.Status 取第一筆，仍保留；per-item 供 app 用）。

### 2. SetStatus 改為「來源狀態感知」（修正混合狀態 bug）
目前 `UPDATE OrdersItem SET ItemStatus=@Status WHERE OrderNO=@OrderNo AND PrintGroup=@KDS_ID` 會把整單所有品項一律改成目標狀態 → 對混合狀態會把 O 打回 W 等。

改為：請求帶 **`FromStatus`**（來源狀態集合），UPDATE 加 `AND ItemStatus IN (<FromStatus>)`：

| 動作 | 目標 Status | FromStatus（只搬這些） | 備註 |
|------|------------|----------------------|------|
| 接單 ProgressOrder | W | j, J | 加點時只動新品項 |
| 完成 FinishOrder | O | W | 設 Finishtime |
| 取餐 CollectedOrder | F | O | |
| 召回 RecallOrder | O | O, F | 重設 Finishtime |

- `FromStatus` 為**選配**：未帶時維持舊行為（更新全單），向後相容。
- 順手清掉現有死碼（`if(req.Status=="preparing") _="W"` 那段，永遠不 match、且 discard）。

## 前端改動（CitrusKDS）

### 1. 資料模型
- `Detail` 新增 `itemStatus`（`@Json "ItemStatus"`）。
- `SetOrderStatusRequest` 新增 `fromStatus`（`@Json "FromStatus"`，選配）。

### 2. 訂單狀態派生
- 新增 `Order.displayStatus()`：依品項狀態派生（有 j→J / 否則有 W→W / 否則 O）。卡片改用派生狀態，不再只看後端 `Status`。
- 新增 `Order.isAddon`：有 j 且有 W/O。
- 新增 `Order.addonItems`：`detail.filter { itemStatus in (j,J) }`。

### 3. 接單流程（ProgressOrder / MainFeatureBtn）
- 送 `fromStatus=["j","J"]`，只升級新品項。
- 列印分流（沿用現有「接單即印」printMode 設定，關閉則不印）：
  - **加點**（isAddon）→ 印 `addonItems` + 標題「加點」。
  - **全新單** → 印整張（現有行為）。

### 4. 完成 / 取餐 / 召回
- 各帶對應 `fromStatus`（見後端表），只搬對應品項。

### 5. 自動接單（autoAccept）
- 現有邏輯只接「整單 NEW」。擴充為：偵測到 `isAddon`（已見過的單冒出新 j）也自動升級 + 印加點單。
- 需調整現有 session 去重（`autoAcceptedThisSession` 以 orderNo 記）：加點要能對「已接過的 orderNo」再次觸發 → 改以「該單仍有未接 j」為條件，升級後 j 消失自然停止。

### 6. 列印
- `EscPosReceiptBuilder` 新增加點模式：只印傳入的品項清單 + 最上方「加點 / Add-on」標題。其餘版面沿用。

## 卡片狀態顯示
- `MainFeatureBtn` 接收派生狀態，沿用先前「按鈕一律反映實際狀態」原則（有 j→新单 / W→处理中 / O→待取餐）。

## 測試重點
1. 單一全新單：接單→W、印整張、無「加點」字樣。
2. W 單加點：卡片跳新單；接單只升級新品項、其餘 W 不動；只印新品項 + 加點標題。
3. O 單加點：卡片跳新單；接單後 O 不被打回 W。
4. 自動接單開：加點自動升級 + 自動印加點單。
5. 連續加點：第二次加點再次觸發。
6. 完成/取餐對混合狀態只搬對應品項（回歸測試 SetStatus）。

## 部署注意
- 後端需在 POS 伺服器（192.168.0.162）重新 build & deploy 才生效。
- 前端 `fromStatus` 選配 → 後端未部署前，前端送了也只是被忽略（舊行為），無 crash；但「只搬對應品項」與「per-item 狀態」要後端到位才完整。建議**後端先部署**再驗證前端加點流程。
