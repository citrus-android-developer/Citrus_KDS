# 自動接單功能（評估中）

廚房常見痛點：員工專注做菜時看不到 KDS 螢幕，新單進來要按一下 Prepare 才能確認收到，造成延遲或漏單。「自動接單」讓新單自動推進到下一個狀態，省去那一下確認。

## 需求摘要

新增設定 `isAutoAcceptEnable: Boolean`（預設關閉）。

| autoAccept | prepareMode | 新單行為 |
|-----------|-------------|---------|
| 關 | 任意 | 維持現狀（J 等員工按 Prepare） |
| 開 | **僅 prepareMode 開啟時生效** | 自動 `J → W`（仍在 Main 頁顯示「製作中」） |
| 開 | prepareMode 關 | **強制 disable autoAccept**（UI 灰掉、避免邏輯衝突） |

> 用戶決策（2026-05-14）：autoAccept 限定 prepareMode 開啟時才能用。理由：prepareMode 關時 J→O 會讓單瞬間進 Served，廚房永遠看不到要做什麼。

## 觸發時機

**下次輪詢接到新單就觸發**（最多 3 秒延遲）。實作在 [[輪詢架構]] `fetchOrders` 成功處理：

```kotlin
// CentralViewModel.kt fetchOrders Result.Success "main" 分支
setState { copy(mainList = result.data, errMsg = null) }

if (prefs.isAutoAcceptEnable && prefs.isPrepareEnable) {
    result.data
        .filter { it.status.uppercase() == NEW }
        .forEach { order ->
            setOrderStatus(order.orderNo, PROGRESSING)
        }
}
```

> 注意：要在 setState 之後才呼叫，避免 race condition。

## 邊界處理

### 1. PrepareMode 中途關閉

**決策（2026-05-14）：強制連動 — prepareMode 關掉時自動把 autoAccept 也關掉**。

實作：[[設定頁]] `PrepareRadio` 觸發 `onPrepareModeChanged(false)` 時：

```kotlin
// CentralViewModel.kt handleEvent
is Event.onPrepareModeChanged -> {
    prefs.isPrepareEnable = event.mode
    if (!event.mode && prefs.isAutoAcceptEnable) {
        prefs.isAutoAcceptEnable = false                 // 強制關閉
        setState { copy(autoAcceptState = false) }       // 同步 UI
    }
    setState { copy(prepareModeState = event.mode) }
}
```

理由：行為直覺、避免「設定頁顯示 autoAccept 開但實際上不會發生任何事」的灰色狀態。重新開 prepareMode 時 autoAccept 預設關閉，需要使用者重新打開（多一步操作但語義清楚）。

UI 配套：`AutoAcceptRadio` 在 prepareMode 關閉時直接 disable/灰掉並提示「先開啟 Prepare Mode」。

### 2. 多筆同時進來

若一次輪詢收到多筆 J，`.forEach { setOrderStatus(...) }` 會連續派發多個 viewModelScope 任務 — 沒有同步保護。後端 `/KDS/SetOrderStatus` 是否支援並發？需確認，否則改 `forEach` 內 `delay(100)` 或改成 sequential。

### 3. setOrderStatus 失敗處理（決策：記憶體 Set）

避免「派發 PROGRESSING 失敗 → 訂單仍是 J → 下次輪詢又派 → 無限重試」的死循環。

`CentralViewModel` 加私有欄位：

```kotlin
private val autoAcceptedThisSession = mutableSetOf<String>()
```

`fetchOrders` 成功分支 main 內：

```kotlin
if (prefs.isAutoAcceptEnable && prefs.isPrepareEnable) {
    result.data
        .filter { it.status.uppercase() == NEW }
        .filter { it.orderNo !in autoAcceptedThisSession }   // 跳過已派發過的
        .forEach { order ->
            autoAcceptedThisSession.add(order.orderNo)        // 先標記
            setOrderStatus(order.orderNo, PROGRESSING)        // 再派發
        }
}
```

**特性**：
- 重啟 App 自動清空 → 持續失敗的單下次冷啟動會再試一次
- `setOrderStatus` 失敗不會自動從 Set 移除 → 不重試（避免無限派發）
- 員工發現某張單卡在 J 仍可手動點 Prepare 補救（按鈕仍有效）

**為什麼不用「重試上限 + 失敗計數」**：實作複雜度提升，且 setOrderStatus 失敗本身已透過 [[資料層錯誤處理]] 的 retry policy 處理（2 次自動重試，間隔 3s/9s）。

### 核心派發邏輯

```kotlin
private fun autoAcceptNewOrders(orders: List<Order>) {
    if (!prefs.isAutoAcceptEnable || !prefs.isPrepareEnable) return
    orders.filter { it.status.uppercase() == NEW }
        .filter { it.orderNo !in autoAcceptedThisSession }
        .forEach { order ->
            autoAcceptedThisSession.add(order.orderNo)   // 先標記再派發
            setOrderStatus(order.orderNo, PROGRESSING)
        }
}
```

### 與評估的差異

UI 反應性改用設定頁 **local state**（仿既有 `isPrinterSelectVisible` 慣例）而非 State 的 `autoAcceptState` — 因 PrepareRadio 本就直接讀 prefs 不走 State，避免再動所有 State 建構子。行為（連動關閉、disable 灰掉）與評估一致。

## 2026-06-03 行為調整：只套用新單（baseline）

需求：設定頁開啟自動接單時，**只套用開啟之後進來的新單，開啟當下已存在的舊 J 單不自動接**。

實作（`CentralViewModel.autoAcceptNewOrders`）：新增 `autoAcceptBaselined` 旗標。
- 開啟（或重啟）後**第一次**輪詢：把當下所有 J 單加入 `autoAcceptedThisSession` 當基準、**不派發**，設 `autoAcceptBaselined=true`。
- 之後輪詢：只對「不在 set 裡」的新單派 W。
- 自動接單**關閉**時 `autoAcceptBaselined=false`，下次開啟重新建立基準。

⚠️ **影響既有決策**：重啟 App 後畫面上既有 J 單也視為「舊單」不自動接 —— 這改變了原評估「重啟清空→失敗單冷啟動重試」的行為。重啟後不再自動補派既有 J 單，需人工處理或等新單進來。


## 2026-06-03 體感優化：樂觀更新（省掉顯示延遲）

問題：自動接單靠輪詢觸發，派 W 後本地不即時更新畫面，要等下一輪輪詢才看到「製作中」（額外 ~1-3 秒延遲）。

改法：新增 `autoAcceptDispatch(orderNo)` 取代直接呼叫 `setOrderStatus`：
- 派發前**樂觀更新** mainList 該單 `status=W`（畫面立刻變製作中）。
- 直接呼叫 `repository.setOrderStatus(W)`；**失敗則把該單 status 改回 J(NEW)** 並顯示錯誤訊息（不自動重試）。
- 輪詢仍會在下一輪用後端真實狀態校正（成功→維持 W、失敗→J），樂觀更新只是讓畫面更快。

效果：新單仍要等輪詢撈到（最多 3 秒）才觸發，但「觸發後 → 畫面變製作中」幾乎即時，省掉原本等下一輪的顯示延遲。
