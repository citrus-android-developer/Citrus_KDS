# 系統模式切換

App 支援兩種模式，由 `prefs.mode` 決定（`0=KDS`, `1=OrderReady`）。模式儲存在 SharedPreferences，App 重啟保留。

## 切換流程（2026-06-04 起：直接導頁）

```
SettingPage 的 ModeRadio onClick
        ▼
event(OnModeChanged(it))   ← 更新 prefs.mode + state.modeState
        ▼
navigateTo(it)             ← 依模式直接導頁（SettingPage→MainActivity setting route）
   if mode==1 → navigate("orderReady")
   else       → navigate("kds")
   { popUpTo("setting"){inclusive=true}; launchSingleTop=true }
        ▼
對應頁顯示（KDS 鎖橫向 / OrderReady 依方向設定）
```

導航結果**不再依賴 async 事件處理時序**，故兩個方向都穩定。

## 舊流程與競態 bug（已修，2026-06-04）

舊版：ModeRadio 一律 `navigateTo("kds")`，再靠 `KdsScreen` 的
`LaunchedEffect(modeState){ if (modeState==1) navigateToOrderReady() }` 補路由。

**Bug**：OrderReady→KDS 時，`KdsScreen` 在 `OnModeChanged` 事件處理前就以**舊 `modeState=1`** 組合 → `LaunchedEffect` 立刻把畫面**彈回 OrderReady** → 使用者覺得「切到 KDS 失靈」。因屬時序競態，故「有時候」才發生。

**修法**：直接導頁（上方）＋**移除** `KdsScreen` 的 `modeState` 彈回（`modeState` 不再用於導頁）。詳見 [[ISSUE-模式切換radio競態失靈]]。

## 切換本身沒有 API 呼叫

`OnModeChanged` 只更新 prefs 與 state，**不打任何 API**。

進入 OrderReady 後才開始輪詢 `/controller/OrdersList`（見 [[OrderReady模式]]）。

## startDestination

`MainActivity.kt:154`：
```kotlin
startDestination = if (prefs.mode == 0) "kds" else "orderReady"
```

冷啟動依 prefs 決定。設定中切換模式時不會冷啟動，改由 `navigateTo(mode)` 直接導到對應 route（不再靠 `KdsScreen` 的 `modeState` 彈回）。
