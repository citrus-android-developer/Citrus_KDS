# 資料模型（VO）

所有與後端交換的 JSON 結構。集中於 `commonData/vo/`。全部用 Moshi `@Json(name="...")` 對應後端 PascalCase 命名。

## 訂單核心

### Order（`Order.kt`）

```kotlin
data class Order(
    @Json(name="Detail")      var detail: List<Detail>,
    @Json(name="OrderTime")   var orderTime: String,
    @Json(name="OrderNo")     var orderNo: String,
    @Json(name="ServiceType") var serviceType: String,
    @Json(name="Status")      var status: String,           // J/W/O/F
    @Json(name="Note")        var note: String? = null,      // 震動機號碼（2026-06-03 新增）
    @Json(ignore=true)        var isVisible: Boolean = true  // 本地動畫旗標
)
```

`isVisible` 是**客戶端用的**（`@Json(ignore=true)`），用於觸發 [[訂單卡片元件]] 的 OK 覆蓋層動畫。

`note`（`@Json(name="Note")`，2026-06-03 新增）= 震動機號碼，來源是後端 Compass_KDS Orders 表的 `Note` 欄位（後端 `OrdersData.Note` 已一併補上撈取與回傳）。[[訂單卡片元件]] 在訂單號旁以括號顯示，空值不顯示。

### Detail（`Order.kt`）

```kotlin
data class Detail(
    GType, CName, EName,                    // 品項類型 (M/G/S)、中英文名
    Flavor?, Addition?,                     // 口味、加料
    GID, GKID, Qty, Price,
    middleDetail: List<Detail>?             // 客戶端組裝的子項目（M/G 下接連 S）
)
```

> `GType` 可能值：M (Main 主品項) / G (Group) / S (Sub 子配料)。`OrderItem` 渲染時迭代 detail，遇到 M/G 時把後續連續 S 收進 `middleDetail`（[[訂單卡片元件]] OrderItem.kt:68-79）。

## 訂單列表查詢

### OrderRequest（`OrderRequest.kt`）

```kotlin
data class OrderRequest(
    @Json(name="KDS_ID")      val kdsId: String,
    @Json(name="OrderStatus") val type: String,    // "main" / "served" / "recall"
)
```

注意：欄位名是 `OrderStatus` 但內容是頁面分類字串（不是訂單 status code 的 J/W/O/F）。命名容易誤導。

## 設計觀察

- **後端 PascalCase / 客戶端 camelCase**：透過 `@Json(name=...)` 一一對應，需小心新增欄位時別漏寫
- **var + mutable**：所有 VO 用 `var`，可改但風險高（已有 Order.detail 在 OrderItem 中被原地修改塞 middleDetail，違反 immutability）
- **status code 散落**：訂單用 J/W/O/F 字元、庫存用 "Available"/"Sold out" 字串、推播狀態用 Int — 不一致
- **可空性混亂**：StockInfo 全 nullable、Order 全 non-null、Request 多為 non-null 但 prefs 為空時會送 ""
