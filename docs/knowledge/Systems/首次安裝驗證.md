# 首次安裝驗證

第一次安裝（`prefs.firstInstall == true`）時強制使用者輸入驗證碼，通過才能進入設定。

## 觸發點

[[設定頁]] `SettingContent.LaunchedEffect(Unit)`（SettingPage.kt:124-128）：

```kotlin
if (prefs.firstInstall) {
    Constants.createCode()
    securityManager = true                  // 開啟 SecurityDialog
}
```

`securityManager` 為 true → 顯示 SecurityDialog，且 `dismissOnBackPress = false` / `dismissOnClickOutside = false`（無法繞過）。

## 驗證碼算法（`Constants.kt:184-199`）

```kotlin
fun createCode() {
    val dateStr = SimpleDateFormat("MMdd").format(Date())
    code = (Math.random() * 99999 + 100001).toInt().toString()     // 100001~199999
    val reversed = StringBuffer(code).reverse().toString()
    finalCode = ((reversed.toInt() + 525111) * 3 + dateStr.toInt()).toString()
}
```

`code` 是顯示給使用者的「題目」，`finalCode` 是「答案」：

1. 取一個 100001~199999 的亂數 = `code`（畫面上顯示）
2. 反轉成字串 → 加 525111 → 乘 3 → 加上日期 MMdd（整數） = `finalCode`

> ⚠️ **設計問題**：演算法寫死在 client，技術上人可以反推。實際上是給內部安裝員看的「半私密」流程，不算真正安全機制。

## SecurityDialog（`widget/SecurityDialog.kt`）

UI：
- 顯示題目 `code`（大字 32sp 藍色）
- TextInputField 輸入答案（Number keyboard）
- Cancel / Verify 兩鈕

驗證邏輯（SecurityDialog.kt:197-204）：

```kotlin
if (state.value.state.text.toString() == finalCode) {
    prefs.firstInstall = false              // 解鎖
    finish()
    onDismissRequest()
} else {
    isCodeCorrect = false                   // icon 變紅色
}
```

通過後 `prefs.firstInstall = false` 永久解鎖此裝置。

## Cancel 路徑（強制離開 App）

點 Cancel → `onVerifyCancel()` → [[KdsScreen容器]] 透過 `ReadyForPage` 把它派發為 `Event.onVerifyCancel` → 設 `state.isVerifyCancel = true` → [[MainActivity入口]] `LaunchedEffect` 偵測到 → `this@MainActivity.finish()` 結束 Activity。

> 設計意圖：沒過驗證者無法使用 App。

## firstInstall 持久化

[[Prefs偏好設定]] `firstInstall: Boolean default true`。一旦 `false` 永遠不再彈 Dialog（除非清資料）。

## 設計觀察

- **演算法寫死在 client** + ApiKey 也寫死（[[依賴注入]]）→ 整套「安全」是儀式性的
- 日期參與計算 → 同一天可重複使用同樣的 `(code, finalCode)`，跨日要重新計算
- Cancel 立刻 finish Activity → 沒有「先看一下不想驗證」的友善路徑
